Maker’s Academy Pre-course Week Two

Today started work on Git. Previouly, I’ve got to say it’s been a little bit frustrating dealing with Git. A lot of strange errors seem to pop up that are difficult to understand as a new comer. The basics are easy but doing more complex things such as going back several commits and merging is tough. The below has massively increased my understanding of Git. I can’t say I’ve had a point of enlightment with Git yet but maybe that will come later.

My favourite resource by far has been http://gitimmersion.com/.

I’ve added my explanations for the git commands below. This is to better aid my own understanding but hopeuflly it will help others too.

Setting Global Settings

git config —global user.name “Jacob Mitchinson” git config —global user.email “email@gmail.com”

These two are immensely useful commands to change your name and email globally for git.

Setting Line Ending Settings

git config —global core.autoclrf input

Your computer uses invisible markers to denote line endings. This can be problematic as Windows and UNIX use different systems to show line endings. So when working on remote repos it’s a good idea to use one format. This line will convert Windows formatted line endings (CLRF) to Unix line endings (LF).

git config —global core.safeclrf true

From StackOverflow: this ‘aborts the conversion of a file if checking out the file will result in a changed and possibly damaged file which would be the case if Notepad was the editor’. I’m not totally satisfied with this answer as I’m not sure how the file would become damaged as a result of the conversion.

Removal

rm -rf .git

The -rf stands for ‘recursive’ (all folders) and the -f forces the removal.

Also ensure to remove files from the staging area like so:

git rm file

This will remove the file itself and from staging in one go.

Viewing Your Commits

If you want to see the commits that have been added we can do so like so:

git log

This one is a nice one too:

git log —pretty=oneline

This shows your commits on a single line along with the corresponding hash.

We can also use other options with —pretty=oneline to define things even further using —max-count, —since, —until, —author, and —all:

git log —pretty=oneline —max-count=3 git log —pretty=oneline —since=’10 minutes ago’ git log —pretty=oneline —until=‘5 minutes ago’ git log —pretty=oneline —all

It was nice to see that when I googled some more info about —pretty=oneline that one of the former (?) teachers at MA, Alex Peattie, wrote an article about this. He also suggested that we could use —before instead of —until and —after instead of —since. I find that much more natural.

I took a lot of this from Git Immersion. A nice command to use is this too:

git log —pretty=format:‘%h %ad | %s%d [%an]’ —graph —date=short

ALIAS SETTINGS

Edit .gitconfig with the following settings to improve your git speed:

[alias] co = checkout ci = commit st = status br = branch
hist = log –pretty=format:‘%h %ad | %s%d [%an]’ –graph –date=short type = cat-file -t dump = cat-file -p

Git Tag

Create tags:

git tag v1

git checkout v1

You can now refer to versions by their names and not have to worry about the hash!

View tags:

git tag

Checkout the tag history using the following:

git hist master —all

Going Back

Remove Tag

git tag -d -all

CLEAR THE STAGE

git reset HEAD hello.rb

Removes staged items.

Revert Changes in Working Directory

To revert changes in the working directory for a file just checkout of the file:

git checkout hello_changed.rb

Undo Commits

git revert HEAD

Reverts commits.

git reset —hard (push hash / tag here)

Resets the branch to the specified commit.

WARNING: Resetting can confuse those on a shared repo.

Edit Commits

Amends the last commit:

git commit —amend -m “amend”

GIT MV

Move files with:

git mv from_here to_here

This is advantageous as git will then automatically at the file to the stage too.

Branches

Creating Branches

git checkout -b newbranch

Show Current Branch

git branch

SHOW REMOTE BRANCHES

git branch -a

 Merging

git merge master

Fixing Conflict

Edit conflicted files

REBASING

git rebase master

The commit tree will get rewritten to show all of the changes from the branches.

Merge should be used for public repos as rebase will confuse those working on the project.

 REMOTE ADD

git remote add shared ../hello.git

CLONE

git clone hello cloned_hello

Clone will help you to copy your repo into a new repo.

GIT REMOTE

Tells you the name of the repos home.

git remote show origin

Show you the location.

GIT FETCH

git fetch

Fetch the change history. NOTE: will not commit. Using git merge origin/master will allow you to get the commits.

GIT PULL

FETCH AND MERGE

 TRACK REMOTE BRANCH

git branch —track greet origin/master

Tracks the greet branch in origin/master.

Bare Basics

git clone —bare hello hello.git

Creates a clone of the hello repo with just the .git information.
