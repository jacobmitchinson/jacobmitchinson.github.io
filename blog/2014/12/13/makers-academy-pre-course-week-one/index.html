Maker’s Academy Pre-course Week One

I began the Makers Academy today which is a 12 week intensive course, focusing on Ruby, Rails, HTML, CSS, and Javascript. This blog is intended to track my progress through the course and to help others that might be trying to learn. I’ve been bringing myself up to speed over the last few months using a number of different resources. The most useful ones are listed below with a brief review:

Books:

Beginning Ruby | Good book but OOP explanations took me a few reads to understand.

The Well Grounded Rubyist | This has a better explanation of some ruby concepts, particularly OOP.

Online:

http://learnrubythehardway.org & learn cli hard way | This was by far the best resource I used for learning the basics of coding. It is not necessarily the best ruby resource, but it is fantastic for teaching you the mindset needed to learn to code. Principally this is because the resource forces you to ask why your code works. If you can forgive the occasionally condescending tone of the author then this is the best place to start.

https://pine.fm/LearnToProgram | Chris Pine probably does a better job at explaining Ruby than the above and the exercises are excellent.

http://rubymonk.com | From the creator of ruby comes Ruby Monk. It’s a good resource but I found the explanations a little bit difficult to understand at times. Perhaps that’s just me.

http://www.theodinproject.com | The Odin Project is a free and open source course that, as it claims, will enable you to become a Web Developer using entirely free resources. Its creator is now running the paid for bootcamp Viking Code School. The resources it uses are good and the ruby exercises rapidly improved my understanding of OOP. I would also highly recommend doing the Web Development 101 part of the course.

http://codeacademy.com - Ruby, HTML, CSS, Javascript tracks | Codeacademy is superb for immediately immersing you in each language. However, its explanations are sometimes frustratingly lightweight and it doesn’t encourage you to branch out from the core materials.

http://www.ee.surrey.ac.uk/Teaching/Unix | Great unix resource.

http://gitimmersion.com | Great git resource with excellent explanations

Things I hated (perhaps just me):

https://www.codeschool.com | I did not like the video format at all and I found the explanations unintuitive.

A quick note: I’ve found that sometimes I’d have to read multiple resources on a similar subject matter in order to appreciate the ins and outs.

Finally, remember this: do not try and understand all of computer science all at once. No seriously, you could potentially spend hours trawling through Wikipedia to understand certain concepts but without understanding the basics you aren’t going to get anywhere. I learnt this the hard way.

Making Life Easier

I consider myself to be quite a patient person, but I’ve also found that sometimes I’ve wanted to throw my computer out the window whilst doing some of these exercises. Seriously, I cannot express enough the importance of taking regular breaks whilst coding. I use the Pomodoro technique to help enforce this using a 50 minute work / 10 minute break cycle. Trust me, it’ll save you time and keep you fresh throughout the day. I like this iPhone app.

Another app I’ve used is ‘Stop, Breathe & Think’ - a meditation app. The short meditations work well with my Pomodoro timings. If you give it a try, you’ll never go back.

Sometimes when learning a new skill you immerse yourself in the task so much that it’s easy to forget about the rest of the world. An app that I’ve used for a little while now, Omnifocus, has saved my bacon more times than I can remember (Omnifocus probably knows though). It uses the Getting Things Done system and will help you to complete projects from ‘make a wizard chess game in ruby’ to harder tasks such as ‘load dishwasher’.

I’m also an avid user of Pocket. It’ll help you get back to resources you use. Just remember to tag the things you like. I also use an IFTTT to send interesting articles to this Tumblr.

Makers Pre-course Week One

Below I have included my explanations for some useful commands that I learnt this week whilst doing the Makers pre-course week one. I like to explain the commands to myself in a concise format. I find that this aids my memory and understanding whilst also providing a handy reference for things I’ve learnt. Further, whilst the below is mostly made up of things I have learnt from the Makers resource, I often find that I use multiple resources to aid my understanding. When I want to look something up quickly, I don’t want to have to remember precisely where I found it.

Command Line

rm -f

This forces the removal of a file even if it is write protected.

rm -i

This provides an interactive screen for deleting files. Especially useful if you are deleting a lot of files.

cat

This concatenates a file which means that it combines files.

cat jenkins.txt

Print the contents to the screen.

cat > cat.txt

This will allow you to edit a file.

cat someText anotherText > combined

This will merge two files and provide you with a “combined” text file.

less longText

Displays the text. ‘q’ to quit ;)

head -3 longText

Displays the first lines within the console. It defaults to 10 but here we specify 3.

Don’t make this mistake and bash your ‘hed’:

tail -3 longText

Tail works in a similar way to head but displays the last lines.

SUPER USEFUL = tail -f log

This command is insanely useful. It allows you to see what is happening with logs (and other messages) on the fly. tail -f /private/var/log/system.log is useful on Macs.

man

RTFM

Streams

Streams are channels that connect to inputs and outputs. Your keyboard is an input stream and your screen is an output stream. Usually there is an error stream.

These are usually abbreviated as STDIN (stream in), STDOUT (stream out), and STDERR.

This is valuable to know because streams can be connected in different ways. Thus:

cat combined.txt | less

Enables us to send the output on the left to less (as earlier) using what is known as a pipe or “|”, Mario style.

This is known as redirection.

Last tip on redirection. The greater than sign will write the output from the input on the left to the input on the right. That’s how we concatenated the following:

cat someText anotherText > new

Finding and listing files

If you are only working with a few files then using > ls is quite handy. However, if you are working with 6,000 files then you are probably going to need to narrow down what you are listing on the screen.

This is where wildcards come in use.

For example:

ls some*

Will list all list all files in the current directory beginning with ‘some’ e.g. someText or moreText.

find . -name “some*” -print

Again, we can combine this with some redirection to save a search to a document. Par exemple, we can use the tilde (~) using ALT + N to go to our movies directory using > cd ~/Movies and then search for our .mkv movies and save them to a file.

cd ~/Movies

find . -name “*.mkv” -print > myMovies.txt

We can also have a peek inside files and search for particular files using Grep:

Globally Search a Regular Expression and Print

grep wally whereswally.txt

OR

grep wally *.txt

Regular expressions (REGEX)

Regular expressions really are fantastic things. They are immensely powerful for defining very specifically what you are looking for from a string. From my knowledge, they maintain a very similar syntax throughout most programming languages. They are hard to remember to begin with but once you know them, you know them.

Here’s a good website to try them out: http://www.regexr.com.

One line crash course:

\w \d \s (word, digit, whitespace)

Grep plays nicely with regular expressions. The following will find any .txt files in the current directory and then send them to grep. Grep will then analyse them to see if they contain any words.

find . -name “*.txt” -print | grep \w

Counting words

This has quite a self-explanatory syntax: wc. You can count the number of words in any file by typing the following:

wc someText

9 492 3004 longText

These are the line, word and character counts for the someText file.

We can use redirection in a similar way and combine this with find & grep:

find . -name “.txt” -print | grep wally | wc

This will show us the number of Wally’s found.

Parameters: -c (show bytes), -l (show lines), -m (show characters), -w (show words)

PERMISSIONS

whoami

There are three main types of classes: “user”, “group” and “other”. There can only be one owner of any one file - the user. Sometimes, groups of users exist to define a class of users. The other class is reserved for those that don’t fall into the user or group class.

There are three main types of permissions: “read”, “write” and “execute”. If we have a look at a directory using:

ls -l

We can view the read / write permissions:

image

The first hyphen here tells us that these are all files. If not, it would say “d” - “d” for directory. Then it begins to tell us the read / write / execute status for the user, group and other. The user comes first. It says that the user has read, write capabilities. The hyphen after the w indicates that there is not an execute option. This gives us “rw-“.

After this we have the group permissions. In the same structure as was defined for the user, we have “r—”. Meaning that the group has read permissions but not write and execute capabilities.

Finally, we have other. We can see that other only has read permissions in the same way as the group.

Changing permissions - change mode

If we want to change permissions, chmod is our friend. Chmod stands for change mode.

chmod u+r someText

Chmod enables us to change the read, write and execute permissions for the user, group and other (all). As above, u stands for user, but this can be interchanged with g(roup) or a(ll). We use + or - minus to indicate what we are adding. As before, we then specify the permissions we wish to change. Above we have changed this to read. Lastly, we specify which file we wish to change: someText.

Shebang

The shebang is a combination of a hash # and an exclamation mark !. It enables us to specify which interpreter we are going to use in a script. This is especially useful if you want to run a script on the command line without specifying what to use.

For example, if we put #! /path/to/ruby at the top of an hello.rb script the console will know that this is a ruby script.

We need to use chmod after this to specify that hello.rb is exectuable.

chmod u+x hello.rb

Now we can run ./hello.rb on the command line and it will run the script.

Note: to make the code run on other systems, it would be wise to change the hello.rb script to: #! /usr/bin/env ruby

A lot of programs use the shebang in order to specify the interpreter to use. For example, irb uses it:

head “which irb”

You can see at the top of the script that irb is using a shebang to specify the interpreter to use.

Environment variables

If you want to have a look at what at the environment you are working with on the command line try > env. It will display everything to the screen that your environment is currently working with.

Echo

echo $HOME

Super useful for telling you your home directory.

And

echo “Hello world!” > hello.txt

 PATH

Path stores the location of paths to your programs and files. Everything that runs on the command line, without you specifying its location, will have a path pre-designated. For example, if you run “ruby” then the computer will search for the PATH locations until it finds the correct file to run. We can see this in action if we run the following:

echo $PATH ruby

We can create or modify our own PATHs using export:

export MA=/path/to/makersprecourse

I created a path for my Makers Precourse files. This means that I can access it by running:

cd $MA

More commonly, environment variables are used to store secret data. For example, say we had an API key for Twitter that we used in our application. We could save this using > export TWITTER_API_KEY=[enterkey].

We could then read this in our ruby script using ENV[TWITTER_API_KEY].

You could then upload to GitHub safe in the knowledge that no one could use your key.

However, environment variables are only saved for the session that you are in.

Navigate to your home directory.

Use > ls -l to display hidden files. You should now see .bash_profile. This is where your terminal settings are stored. If we want to permanently add the environment variables we have set then we will have to save them here.

echo “export MA=/path/to/location” » ~/.bash_profile

Examine the bash file now and you should see the changes.

VIM

In the past I’ve ran a mile every time VIM cropped up and instead I used nano. But VIM is actually very easy to use once you know a few tricks.

For a start, VIM has two main modes: command mode and edit mode.

Let’s boot up vim:

vim newFile

To start with you will be in command mode. To switch to edit mode press “i” for insert. Then you can type and use two other commands to help you. “O” will enable you to get to the next line and “dd” will delete a line.

Now press ESC.

You are now back in command mode. Easy, huh?

To save, type :w which stands for write. Alternatively, type :q to exit or :q! to exit without saving.
